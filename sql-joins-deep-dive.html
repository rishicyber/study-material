02 — Join Types

INNER JOIN

Returns only matching rows from both tables
The most common join. If there's no match on the join condition, the row is excluded entirely from the result set.

A ∩ B

Only rows where employees.dept_id = departments.id exist in both tables are returned. Eve (NULL dept_id) and HR (no employees) are both excluded.

SQL Query

SELECT
  e.id,
  e.name        AS employee,
  d.name        AS department,
  d.location,
  e.salary
FROM   employees e
INNER JOIN departments d
  ON e.dept_id = d.id
ORDER BY e.id;

⟶ Result Set (4 rows)

1AliceEngineeringNYC90000

Notice: Eve (no dept) and HR (no employees) are completely absent. 5 employees → 4 rows.

LEFT JOIN

All rows from the left table, matched rows from the right
Every row in the left (first) table is preserved. Where no match exists in the right table, NULL values fill the right-side columns.

All of A

All 5 employees appear. Eve has no matching department, so department columns are NULL. HR is still excluded (it's on the right side with no match).

SQL Query

SELECT
  e.id,
  e.name        AS employee,
  d.name        AS department,
  d.location
FROM   employees e
LEFT JOIN departments d
  ON e.dept_id = d.id
ORDER BY e.id;

Finding unmatched rows

A very common pattern: use LEFT JOIN + WHERE IS NULL to find orphaned records.

-- Find employees with no department assigned
SELECT e.id, e.name
FROM   employees e
LEFT JOIN departments d ON e.dept_id = d.id
WHERE  d.id IS NULL;

RIGHT JOIN

All rows from the right table, matched rows from the left
The mirror of LEFT JOIN. Every row in the right (second) table is kept. Right JOINs are rare in practice — most developers flip the table order and use a LEFT JOIN instead.

All of B

All 4 departments appear. HR has no employees, so employee columns show NULL. Eve (no dept) is excluded because she has no matching department row.

SQL Query

SELECT
  d.id          AS dept_id,
  d.name        AS department,
  e.name        AS employee
FROM   employees e
RIGHT JOIN departments d
  ON e.dept_id = d.id
ORDER BY d.id;

⟶ Result Set (5 rows)

Tip: A RIGHT JOIN B is always equivalent to B LEFT JOIN A. Prefer LEFT JOIN for consistency — most style guides ban RIGHT JOIN entirely.

FULL OUTER JOIN

All rows from both tables, NULLs where no match
The union of LEFT JOIN and RIGHT JOIN. Every row from both tables appears — matched rows are combined, unmatched rows get NULLs on the missing side. Not supported in MySQL (use UNION workaround).

A ∪ B

Both Eve (no dept) and HR (no employees) appear, each with NULLs on the unmatched side. Zero data is lost.

SQL Query (PostgreSQL / SQL Server / SQLite)

SELECT
  e.name  AS employee,
  d.name  AS department
FROM   employees e
FULL OUTER JOIN departments d
  ON e.dept_id = d.id
ORDER BY e.id;

MySQL Workaround (UNION)

-- MySQL doesn't support FULL OUTER JOIN natively
SELECT e.name AS employee, d.name AS department
FROM employees e LEFT JOIN departments d ON e.dept_id = d.id
UNION
SELECT e.name, d.name
FROM employees e RIGHT JOIN departments d ON e.dept_id = d.id;

CROSS JOIN

Every row from A paired with every row from B
Produces the Cartesian product — no join condition is needed or used. 5 employees × 4 departments = 20 rows. Use carefully; on large tables this can be explosive.

SQL Query

SELECT
  e.name  AS employee,
  d.name  AS department
FROM   employees e
CROSS JOIN departments d
ORDER BY e.id, d.id;

Real-world use: generating combinations

-- Generate all possible size × color product SKUs
SELECT
  s.size_name,
  c.color_name,
  CONCAT(s.size_code, '-', c.color_code) AS sku
FROM   sizes  s
CROSS JOIN colors c;
Warning: Never accidentally write a cross join by forgetting the ON clause in an older-style comma join (FROM A, B). Always use explicit JOIN syntax.

SELF JOIN

A table joined to itself using aliases
Not a separate SQL keyword — it's any join where both sides reference the same table. Essential for hierarchical data like org charts, bill-of-materials, or friend networks.

03 — Summary

Join Comparison

A quick reference for choosing the right join type.

Join Type Left Unmatched? Right Unmatched? Rows in Result Use When…
INNER JOIN ✗ Excluded ✗ Excluded ≤ min(|A|, |B|) You only want rows with data on both sides
LEFT JOIN ✓ Kept (NULLs) ✗ Excluded ≥ |A| You need all left rows + optional right data
RIGHT JOIN ✗ Excluded ✓ Kept (NULLs) ≥ |B| You need all right rows + optional left data
FULL OUTER JOIN ✓ Kept (NULLs) ✓ Kept (NULLs) ≥ max(|A|, |B|) You can't afford to lose any rows from either side
CROSS JOIN ✓ All pairs ✓ All pairs |A| × |B| Generating all combinations (sizes, dates, SKUs…)
SELF JOIN Depends on join type used Varies Hierarchical/recursive data in one table

04 — Performance

Performance Tips

Joins are where most slow queries hide. These practices dramatically reduce execution time.

01

Index your join columns

Always index foreign keys. Without an index, the DB scans every row for each match (O(n²) nightmare).

02

Filter early with WHERE

Apply WHERE conditions before the join using subqueries or CTEs to reduce the working set first.

03

SELECT only needed columns

Avoid SELECT * across joins — fetch only the columns you need to minimize data movement.

04

Use EXPLAIN / EXPLAIN ANALYZE

Always examine the query plan. Look for Seq Scans on large tables — those are index opportunities.

05

Watch out for fan-out

Joining on a non-unique column multiplies rows. Aggregate first, then join to avoid inflated result sets.

06

Prefer INNER over OUTER when possible

INNER JOINs are easier to optimize. Use OUTER only when you genuinely need unmatched rows.

Indexing Example

-- Create index on the foreign key BEFORE running heavy joins
CREATE INDEX idx_employees_dept_id
  ON employees(dept_id);

-- Check the query plan
EXPLAIN ANALYZE
SELECT e.name, d.name
FROM  employees e
JOIN  departments d ON e.dept_id = d.id;
-- Look for "Index Scan" instead of "Seq Scan"
Golden rule: Any column that appears in an ON clause or a WHERE clause on a large table should be indexed. Start there and use EXPLAIN to verify the optimizer picked it up.
