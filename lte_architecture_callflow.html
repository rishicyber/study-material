LTE Overview & Standards

Long-Term Evolution (LTE) is a 4th-generation (4G) wireless broadband standard developed by the 3rd Generation Partnership Project (3GPP) beginning with Release 8 (2008). It represents a complete redesign of mobile network architecture, moving away from the circuit-switched heritage of 2G/3G toward an entirely packet-switched, IP-based system.

The name "Long-Term Evolution" reflects the intent to create a technology platform that could evolve over time — and indeed it has, through Releases 9, 10 (LTE-Advanced), 11, 12, 13, 14, and 15, each adding significant capabilities including carrier aggregation, massive MIMO, device-to-device (D2D) communication, and narrowband IoT (NB-IoT).

Core Design Objectives

PERFORMANCE

Peak Data Rates

Downlink: 300 Mbps (Release 8), up to 1 Gbps (LTE-Advanced). Uplink: 75 Mbps. Achieved via OFDMA and MIMO antenna techniques.

LATENCY

Low Latency

User-plane latency under 5 ms one-way. Control-plane latency (idle to active) under 100 ms. Critical for real-time services and VoLTE.

SPECTRUM

Flexible Bandwidth

Scalable from 1.4 MHz to 20 MHz channel bandwidths. Operates across multiple frequency bands (FDD and TDD modes).

ARCHITECTURE

Flat IP Architecture

Eliminated RNC and introduced the flat all-IP Evolved Packet System (EPS), simplifying the network and reducing cost.

Fundamental Architecture Principles

1. Flat Architecture

Unlike UMTS which had a hierarchical BSC/RNC controlling multiple base stations, LTE distributes intelligence to the eNodeB. There is no centralized radio network controller. This flattening reduces signalling delay and simplifies the network hierarchy.

2. Separation of Control and User Planes

The control plane handles signalling (authentication, session management, mobility) while the user plane carries actual data traffic. These planes are separated both logically and often physically, allowing each to be scaled independently. In the core, this manifests as separate MME (control) and SGW/PGW (user-plane) nodes.

3. All-IP Bearer Model

Data is always carried as IP packets. Connections are modeled as "EPS bearers" — logical tunnels between the UE and the PDN Gateway using GTP (GPRS Tunnelling Protocol) encapsulation over the S1-U and S5/S8 interfaces.

4. Mobility via X2 and S1 Handovers

LTE supports seamless mobility through handovers. The X2 interface enables direct eNB-to-eNB handovers (faster, less core involvement), while S1-based handovers go through the MME/SGW and are used when X2 is unavailable.

Cell Types and Deployment Scenarios

MACRO CELL

Macro eNB

Standard outdoor base station. Coverage radius 1–30 km depending on frequency band and terrain. Typically 20–40W RF output per sector. 3-sector configuration is most common.

SMALL CELL

Pico / Femto eNB

Low-power indoor/outdoor cells for capacity hotspots. Femtocells (Home eNBs / HeNB) serve 1–8 users; picocells serve small buildings. Connect via broadband backhaul (DSL, fiber).

RELAY NODE

Relay Node (RN)

Introduced in Release 10. Wirelessly backhauled from a Donor eNB (DeNB) via the Un interface. Extends coverage to areas without wired backhaul.

HET-NET

Heterogeneous Networks

Combination of macro and small cells on the same or different frequencies. Requires inter-cell interference coordination (eICIC) and advanced scheduling.

Carrier Aggregation (LTE-Advanced)

Starting with Release 10, LTE-Advanced introduced Carrier Aggregation (CA), allowing a UE to simultaneously receive/transmit on multiple component carriers (CCs), each up to 20 MHz wide, for a combined bandwidth of up to 100 MHz (5 CCs). This dramatically increases peak throughput.

eNodeB Deep Dive

The eNodeB (eNB) is the fundamental building block of the LTE radio access network. It is a base station that contains all the intelligence previously split between a 3G base station (NodeB) and its Radio Network Controller (RNC). This consolidation simplifies the architecture and reduces latency.

eNB Functions — Radio Resource Management (RRM)

The eNB is responsible for all RRM decisions, including:

SCHED
Downlink/Uplink Scheduling
The eNB scheduler allocates physical resource blocks (PRBs) to UEs every 1 ms Transmission Time Interval (TTI). Scheduling algorithms consider channel quality (CQI reports from UEs), QoS requirements, buffer status, and fairness. Common algorithms: Round Robin, Proportional Fair, Maximum Throughput.
HARQ
Hybrid ARQ (HARQ)
LTE uses HARQ with Incremental Redundancy (IR) for link adaptation. The UE/eNB retransmits lost packets with additional redundancy, combining multiple transmissions for improved decoding. HARQ operates at Layer 1, with Round-Trip Time of ~8 ms.
AMC
Adaptive Modulation and Coding (AMC)
Based on Channel Quality Indicator (CQI) feedback from the UE, the eNB selects the most appropriate Modulation and Coding Scheme (MCS): QPSK (low SNR), 16-QAM (medium SNR), 64-QAM (high SNR), or 256-QAM (LTE-Advanced Pro). This maximizes spectral efficiency.
RRC
Radio Resource Control (RRC)
The RRC protocol manages UE state (RRC_IDLE, RRC_CONNECTED), connection establishment, reconfiguration, and release. RRC messages carry system information, connection setup, measurement configuration, and handover commands.
ICIC
Inter-Cell Interference Coordination (ICIC)
Neighboring eNBs coordinate frequency resource usage via X2 messages (Relative Narrowband TX Power / RNTP, High Interference Indication / HII) to manage interference at cell edges. Enhanced ICIC (eICIC) in Rel-10 adds Almost Blank Subframe (ABS) for HetNets.

Evolved Packet Core (EPC)

The Evolved Packet Core (EPC) is the heart of the LTE system, handling all control-plane and user-plane functions for the mobile network. It replaces the GPRS Core and 3G CN with a unified, all-IP architecture. The EPC consists of several distinct functional entities, each with specialized roles.

MME — Mobility Management Entity

The Mobility Management Entity (MME) is the key control-plane node in the EPC. It handles all signalling for UE attachment, authentication, session management, and mobility. The MME does not process user-plane data — it only handles control messages.

MME Functions

NAS
NAS Signalling Termination
The MME terminates the Non-Access Stratum (NAS) protocol from the UE. NAS messages are transparently carried by RRC at the radio layer (via eNB) and terminated at the MME. NAS handles EMM (EPS Mobility Management) and ESM (EPS Session Management) procedures.
AUTH
Authentication & Security
The MME initiates AKA (Authentication and Key Agreement) with the HSS to authenticate the UE. It generates security keys (KeNB for RRC/UP encryption, NAS keys for NAS protection) and manages ciphering/integrity algorithms (AES, SNOW 3G, ZUC).
IDLE
Idle Mode Tracking
In RRC_IDLE mode, the MME tracks UE location at the Tracking Area (TA) level. A Tracking Area List (TAL) is assigned to the UE so it doesn't need to update for every cell change within the list. When data arrives for an idle UE, the MME triggers paging across all eNBs in the UE's tracking area.
SGSN
S3 Interface (SGSN Interworking)
For inter-RAT handovers (LTE ↔ 3G/2G), the MME interworks with the 3G SGSN via the S3 interface using GTPv2-C. Context transfers allow seamless handovers between LTE and legacy networks.
LOAD
MME Pool and Load Balancing
Multiple MMEs can form a pool (S10 interface). When a UE attaches, the eNB selects an MME from the pool using a load-balancing algorithm. Intra-pool handovers use S10 for context transfer between MMEs.

Serving Gateway (SGW)

The Serving Gateway (SGW) is the user-plane anchor for LTE. It routes and forwards data packets between the E-UTRAN and the PGW. The SGW is also the mobility anchor for intra-LTE handovers and inter-3GPP-RAT mobility.

SGW Primary Functions

ROUTING

Data Path Routing

Maintains GTP-U tunnels on both the S1-U interface (toward eNB) and S5/S8 interface (toward PGW). Acts as the forwarding node, switching GTP packets between these interfaces per bearer.

MOBILITY

Handover Anchor

During X2 handover, the SGW maintains the existing S5/S8 tunnel to the PGW while the S1-U tunnel is switched to the target eNB. This prevents data loss and avoids PGW reanchoring for local mobility.

CHARGING

Lawful Intercept & Charging

Performs packet counting and charging data record (CDR) generation for offline charging. Also interfaces with the Packet Data Network (PDN) for lawful interception requirements on the S1-U path.

BUFFER

Downlink Buffering

When a UE is in RRC_IDLE state, the SGW buffers incoming downlink packets and triggers paging via the MME. This ensures no packets are lost while the UE reestablishes its radio connection.

PDN Gateway (PGW)

The PDN Gateway (PGW) is the topmost anchor point in the EPC user plane. It connects the EPC to external Packet Data Networks (PDNs) such as the Internet, corporate intranets, or the IMS (IP Multimedia Subsystem) for VoLTE. The PGW is the persistent IP anchor — regardless of how the UE moves, its IP address and PGW association remain stable.

PGW Functions

IPAM
IP Address Allocation
The PGW allocates IPv4 and/or IPv6 addresses to UEs upon PDN connection establishment. Addresses may be statically assigned per subscriber (from HSS) or dynamically assigned from a pool using DHCP or internal allocation. The PGW acts as the UE's default gateway for all data traffic.
TFT
Traffic Flow Templates (TFT) and Packet Filtering
The PGW applies packet filters (TFTs) to map IP flows to specific EPS bearers. A TFT defines matching criteria: source/destination IP address, port ranges, protocol type, and ToS/DSCP. This enables QoS differentiation — for example, routing VoIP RTP streams to a dedicated GBR bearer while HTTP goes on the default bearer.
PCEF
Policy and Charging Enforcement Function (PCEF)
The PGW hosts the PCEF, which enforces policies received from the PCRF via the Gx interface. These include QoS enforcement (maximum bitrate, guaranteed bitrate), gating control (allow/block specific flows), and online/offline charging triggers. The PGW is where per-flow policing and metering happens.
NAT
NAT and IPv4/IPv6 Translation
When UE connectivity is IPv4 but the network uses IPv6 (or vice versa), the PGW can perform NAT64/DNS64 translation. In large networks with limited public IPv4 space, the PGW may perform NAPT (Network Address and Port Translation), mapping multiple UEs to fewer public IPs.

Multiple PDN Connections

A UE can simultaneously maintain connections to multiple PDNs through a single PGW or multiple PGWs. For example, a UE might have one PDN connection to the Internet and another to the IMS network for VoLTE. Each PDN connection has its own IP address and set of bearers.

HSS — Home Subscriber Server

The Home Subscriber Server (HSS) is the central database of the LTE core network. It stores all subscriber-related information and is the authoritative source for authentication credentials, subscription data, and roaming permissions. The HSS is the evolution of the HLR (Home Location Register) used in 2G/3G networks.

S6a Interface (MME ↔ HSS)

The S6a interface uses the Diameter protocol (RFC 3588 / 3GPP TS 29.272). Key Diameter command codes used on S6a:

  • AIR/AIA Authentication-Information-Request/Answer — MME requests authentication vectors for UE.
  • ULR/ULA Update-Location-Request/Answer — MME registers UE location, HSS returns subscription data.
  • CLR/CLA Cancel-Location-Request/Answer — HSS cancels UE registration at old MME during roaming.
  • IDR/IDA Insert-Subscriber-Data-Request/Answer — HSS pushes updated subscription to MME.
  • DSR/DSA Delete-Subscriber-Data-Request/Answer — HSS removes subscription data from MME.
  • PUR/PUA Purge-UE-Request/Answer — MME informs HSS when UE is purged (e.g., long inactivity).
  • RSR/RSA Reset-Request/Answer — HSS notifies MME of reset event.
  • NOR/NOA Notify-Request/Answer — MME notifies HSS of UE reachability.

PCRF — Policy and Charging Rules Function

The Policy and Charging Rules Function (PCRF) is the intelligence center for QoS and charging policy in the EPC. It generates Policy and Charging Control (PCC) rules that govern how the PGW (PCEF) handles individual IP flows — determining their allowed bitrates, charging methods, and gating policies.

PCRF Interfaces

  • Gx PCRF ↔ PGW (PCEF). Delivers PCC rules for session and flow management. Uses Diameter.
  • Rx PCRF ↔ IMS/P-CSCF. IMS signals media parameters (SDP) to PCRF, which authorizes corresponding QoS bearers for VoLTE.
  • Sd PCRF ↔ TDF (Traffic Detection Function). Deep Packet Inspection (DPI) results inform PCRF policy decisions.
  • Sy PCRF ↔ OCS (Online Charging System). PCRF consults OCS for spending limits in pre-paid sessions.
  • S9 H-PCRF ↔ V-PCRF. In roaming scenarios, the home PCRF coordinates with the visited PCRF.

Protocol Stack

Radio Protocol Layers

PHY — Physical Layer

The Physical Layer (PHY) handles all radio transmission and reception. LTE uses OFDMA (Orthogonal Frequency Division Multiple Access) in the downlink and SC-FDMA (Single-Carrier FDMA) in the uplink. Key PHY concepts:

ConceptDescription
Resource Block (RB)Basic scheduling unit: 12 subcarriers × 7 OFDM symbols = 1 slot (0.5 ms). 2 slots = 1 subframe = 1 TTI = 1 ms.
Subcarrier Spacing15 kHz standard spacing. Each RB is 180 kHz wide (12×15 kHz). 20 MHz channel = 100 RBs.
Frame Structure10 ms radio frame → 10 subframes → 20 slots. FDD: all subframes bidirectional. TDD: configurable UL/DL ratio.
Reference SignalsCRS (Cell-Specific RS) for channel estimation. DM-RS (Demodulation RS) for PDSCH. CSI-RS (Channel State Info RS) for feedback.
PDSCHPhysical Downlink Shared Channel — carries user data. Modulation: QPSK, 16/64/256-QAM.
PUSCHPhysical Uplink Shared Channel — carries UE uplink data using SC-FDMA. Same MCS options as DL.
PDCCHPhysical Downlink Control Channel — carries Downlink Control Information (DCI) for scheduling grants.
PUCCHPhysical Uplink Control Channel — carries UCI: CQI, HARQ-ACK, Scheduling Request (SR).
PRACHPhysical Random Access Channel — UE uses this for initial random access using Zadoff-Chu sequences.

MAC — Medium Access Control

The MAC layer sits above PHY and manages: multiplexing of logical channels, HARQ operation, random access procedure, scheduling request, and BSR (Buffer Status Reporting). The MAC communicates with the PHY via transport channels (DL: DL-SCH, BCH, PCH; UL: UL-SCH, RACH).

RLC — Radio Link Control

RLC operates above MAC and provides three modes:

MODE

TM — Transparent Mode

No RLC header added. Used for BCCH (broadcast) and CCCH (initial connection). No segmentation or ARQ.

MODE

UM — Unacknowledged Mode

One-way delivery without ARQ. Used for real-time services like VoIP where retransmission would be too late. Supports segmentation and reordering.

MODE

AM — Acknowledged Mode

Full ARQ with STATUS PDUs confirming successful delivery. Used for TCP data where reliability is required. Supports segmentation, reassembly, and reordering.

PDCP — Packet Data Convergence Protocol

PDCP sits above RLC and provides: IP header compression (ROHC — Robust Header Compression), ciphering of user-plane data and RRC signalling, integrity protection of RRC signalling, and in-sequence delivery/reordering during handovers. During handover, PDCP provides PDCP SDU reordering so packets are delivered to the higher layers in sequence despite the disruption caused by the handover.

Step-by-Step Explanation

STEP 1–2
Random Access (RACH)
The UE initiates random access on the PRACH. It sends a preamble (one of 64 Zadoff-Chu sequences from PRACH configuration). The eNB responds with a Random Access Response (RAR) containing: timing advance (TA) to synchronize uplink, temporary C-RNTI for identification, and UL grant for the RRC Connection Request.
STEP 2
RRC Connection Setup
The UE sends an RRC Connection Request message (on CCCH) containing its identity (S-TMSI if known, or random value) and establishment cause (e.g., "mo-Signalling"). The eNB responds with RRC Connection Setup, configuring initial SRB1 (Signalling Radio Bearer 1) with RLC-AM. The UE completes setup with RRC Connection Setup Complete, which carries the NAS Attach Request in a container.
STEP 3
NAS Attach Request
The eNB extracts the NAS Attach Request from the RRC message and forwards it to the MME via S1AP INITIAL UE MESSAGE. The Attach Request contains: UE identity (IMSI or old GUTI), UE network capability (supported ciphering/integrity algorithms, voice domain preference), PDN connectivity request (requested APN, PDN type), and last visited TAI.
STEP 4–5
Authentication Vector Request
The MME sends a Diameter Authentication-Information-Request (AIR) to the HSS over S6a, requesting authentication vectors (AVs) for the IMSI. The HSS generates Authentication Vectors using the MILENAGE algorithm with the subscriber's secret key K and a random number RAND. Each AV contains: RAND (128-bit random), XRES (expected response), AUTN (authentication token), and KASME (session key). The HSS returns multiple AVs (typically 5) in the AIA response.
STEP 6–7
Authentication (AKA)
The MME selects one AV and sends the Authentication Request NAS message to the UE containing RAND and AUTN. The UE's SIM verifies the AUTN using its own copy of K — this authenticates the network to the UE (mutual authentication). The SIM computes RES (response) and returns it in the Authentication Response. The MME compares RES with XRES from the AV — if they match, the UE is authenticated. Both the UE and MME now independently derive the same KASME, from which all subsequent security keys are derived.
STEP 8
Security Mode Command
The MME selects NAS security algorithms and sends a NAS Security Mode Command, which is integrity-protected using the derived NAS integrity key (KNASint). The UE activates NAS security and responds with Security Mode Complete. From this point, all NAS messages are ciphered and integrity-protected.
STEP 8–9
Update Location (HSS Registration)
The MME sends an Update-Location-Request (ULR) to the HSS to register itself as the UE's serving MME. The HSS responds with ULA containing the subscriber's subscription data: allowed APNs, QoS profiles per APN, subscribed UE-AMBR (Aggregate Maximum Bit Rate). The HSS also sends a Cancel-Location-Request (CLR) to the previous MME (if any) to deregister the UE there.
STEP 10–15
Default Bearer Establishment
The MME sends a Create Session Request to the SGW (GTPv2-C, S11 interface) containing: IMSI, APN, PDN type (IPv4/IPv6/IPv4v6), serving network, RAT type, UE-AMBR, and default bearer QoS (QCI=9, ARP). The SGW forwards a Create Session Request to the PGW (S5/S8). The PGW interacts with the PCRF (CCR-Initial/CCA-Initial on Gx) to obtain PCC rules. The PGW allocates an IP address for the UE and returns a Create Session Response with the allocated IP and TEID. This response propagates back through SGW to MME.
STEP 16
Attach Accept
The MME sends an Attach Accept (NAS) message to the eNB via S1AP INITIAL CONTEXT SETUP REQUEST. This S1AP message also includes: UE security context (KeNB and selected algorithms), E-RAB (Radio Access Bearer) parameters (QCI, ARP, GBR if any), and SGW S1-U TEID and IP address. The eNB uses this to set up the radio bearer and S1-U GTP tunnel to the SGW.
STEP 17
RRC Connection Reconfiguration
The eNB sends an RRC Connection Reconfiguration message to the UE, adding SRB2 (Signalling Radio Bearer 2, for NAS integrity-protected messages) and DRB1 (Data Radio Bearer, the actual user data bearer). The message includes radio bearer configuration: PDCP, RLC, MAC parameters, and logical channel configuration. The UE applies the configuration and responds with RRC Connection Reconfiguration Complete. The NAS Attach Accept is delivered to the UE within this RRC message.
STEP 18
Attach Complete
The UE sends a NAS Attach Complete message to the MME (carried in RRC UL Information Transfer). This acknowledges the Attach Accept and completes the attach signalling. The UE now has a GUTI assigned by the MME for future identification.
STEP 19–20
Modify Bearer (S1-U Path Completion)
The MME sends a Modify Bearer Request to the SGW containing the eNB's S1-U TEID and IP address. The SGW now knows where to forward downlink user data (to the eNB), completing the end-to-end data path: Internet → PGW → SGW → eNB → UE. The SGW responds with Modify Bearer Response. The UE can now send and receive data.

EPS-AKA (Authentication and Key Agreement)

LTE uses the EPS-AKA procedure, derived from UMTS-AKA but enhanced with improved key derivation and separation:

StepAction
1. HSS generates AVHSS generates RAND (128-bit random), computes MAC-A and XRES using MILENAGE with K and SQN. Produces AUTN = SQN ⊕ AK || AMF || MAC-A.
2. MME sends challengeMME sends {RAND, AUTN} to UE in Authentication Request.
3. UE verifies networkUE SIM checks AUTN to verify the network is authentic (this is the mutual authentication step — 2G lacked this). If valid, UE computes RES and sends it back.
4. MME verifies UEMME compares RES (from UE) with XRES (from HSS). Match = UE authenticated.
5. Key derivationBoth sides independently compute KASME from {CK, IK, SQN, PLMN ID}. From KASME, all other keys are derived using KDF (HMAC-SHA-256 based).

Ciphering Algorithms (EEA)

  • EEA0 NULL algorithm — no ciphering. Used in emergency calls or when explicitly configured.
  • EEA1 SNOW 3G based stream cipher. Optimized for hardware implementation.
  • EEA2 AES-CTR mode (128-bit key). Widely deployed, strong security.
  • EEA3 ZUC stream cipher. Developed in China, standardized for 3GPP.

Integrity Algorithms (EIA)

  • EIA0 NULL — no integrity. Not allowed for normal operation (only emergency bearers).
  • EIA1 SNOW 3G based MAC-I computation.
  • EIA2 AES-CMAC (128-bit key). Most commonly deployed.
  • EIA3 ZUC based integrity.

Note on User-Plane Integrity: In LTE Release 8–14, user-plane data is only ciphered (not integrity-protected) at the PDCP layer. NAS and RRC signalling are both ciphered AND integrity-protected. Release 15 introduced optional UP integrity protection for specific scenarios.

Handover Procedures

Handover (HO) is the process of transferring an active connection from one cell to another while maintaining service continuity. LTE supports several handover types depending on the relationship between source and target nodes.

X2 Handover (Intra-LTE, Same MME/SGW)

X2 handover is the most common type. The two eNBs communicate directly via the X2 interface, minimizing core network involvement and achieving the fastest handover (~50 ms interruption).

Key X2 Handover Steps:

Measurement Reports: Based on RRC measurement configuration, the UE measures reference signal received power (RSRP) and RSRQ from neighboring cells and reports them to the serving eNB. The A3 event (neighbor becomes stronger than serving by offset) typically triggers handover.

Handover Decision: The source eNB decides to hand over the UE, selects the target cell, and sends an X2AP Handover Request to the target eNB with the UE's context (security context, current QoS configuration, current C-RNTI).

Target Preparation: The target eNB admits the UE (admission control), configures radio resources, and responds with an X2AP Handover Request Acknowledge containing the RRC Connection Reconfiguration message (the HO Command) that the source will forward to the UE.

HO Command & Execution: The source eNB forwards the HO Command to the UE in an RRC Connection Reconfiguration message. The UE immediately applies the configuration, disconnects from the source, and performs RACH on the target cell.

SN Status Transfer: The source eNB sends an SN (Sequence Number) Status Transfer to the target, reporting which PDCP SDUs have been acknowledged and which are in-flight — this enables the target to resume from the correct point without packet loss or duplication.

Path Switch: The target eNB notifies the MME via S1AP Path Switch Request. The MME updates the SGW to switch the S1-U DL tunnel to the target eNB. The SGW confirms, and any buffered downlink packets are now sent to the target eNB.

S1 Handover (Inter-MME or SGW Change)

When X2 is unavailable or an MME/SGW change is needed (e.g., large geographic move), the S1 handover procedure is used. This involves more core network signalling but still achieves seamless handover. The source MME and target MME coordinate via S10, and the SGW may be relocated via S11 signalling.

Inter-RAT Handover (LTE ↔ 3G/2G)

For areas with LTE-to-WCDMA or LTE-to-GSM coverage transitions, inter-RAT handovers are performed. The MME interworks with the SGSN via S3 (GTPv2-C), transferring the UE's bearer context and security information. The UE performs a connected mode handover from LTE to WCDMA, or the network may use redirection (which is less seamless) for LTE to GSM.

Bearer Modification

Bearers can be modified after creation to change QoS parameters. This is used when:

A video call upgrades from audio-only (modify bearer from QCI=1 audio to add QCI=2 video). A fair-use policy throttles a user's AMBR mid-session. The PCRF updates PCC rules (e.g., at midday peak hours, non-GBR bearers are rate-limited). The modification procedure mirrors the creation procedure but uses Modify Bearer / Update Bearer commands on the S11 and S5/S8 interfaces.

Idle Mode & Paging

When a UE has no active data transfer, it transitions to RRC_IDLE to conserve battery power. In idle mode, the UE disconnects from the radio but remains registered with the network (EMM-REGISTERED). The network can still page the UE when incoming data arrives.

Idle Mode Behavior

DRX
Discontinuous Reception (DRX)
In idle mode, the UE uses DRX cycles (ranging from 320 ms to 2.56 seconds for LTE paging DRX) to wake up periodically, check for paging messages, and sleep again. This dramatically reduces battery consumption — a UE may only be "awake" for a few milliseconds per DRX cycle.
TAU
Tracking Area Update (TAU)
The UE monitors which cells it camps on and performs a Tracking Area Update (TAU) when it moves to a cell in a new Tracking Area not in its assigned TAL (Tracking Area List). The TAU informs the MME of the UE's new location so paging can be directed to the correct area. Periodic TAUs are also configured (e.g., every 54 minutes) to confirm UE is still reachable.

Connected Mode DRX (C-DRX)

Even in RRC_CONNECTED state, LTE supports Connected-mode DRX (C-DRX) where the UE sleeps between scheduled transmissions. C-DRX parameters (DRX cycle, on-duration timer, inactivity timer) are configured by the eNB and balance latency (shorter cycle) against power saving (longer cycle). A typical C-DRX cycle for a smartphone might be 80 ms with a 4 ms on-duration.

MIMO & Physical Layer

Multiple Input Multiple Output (MIMO) is a foundational technology in LTE, using multiple antennas at both the transmitter and receiver to increase spectral efficiency and reliability without additional spectrum.

CSI Feedback

Channel State Information (CSI) feedback is the mechanism by which the UE reports radio channel quality to the eNB. It consists of:

  • CQI Channel Quality Indicator (4 bits, 0–15). Maps to an MCS. The highest MCS supportable at ≤10% BLER.
  • PMI Precoding Matrix Indicator. Tells the eNB which precoding matrix (from a codebook) best matches the channel.
  • RI Rank Indicator. Number of independent spatial streams the channel can support (1–8).

Uplink: SC-FDMA

The LTE uplink uses Single-Carrier FDMA (also called DFT-Spread OFDMA, DFT-S-OFDMA). Unlike pure OFDMA, SC-FDMA has a lower Peak-to-Average Power Ratio (PAPR), which is critical for UE battery life since the power amplifier can operate more efficiently. The UE performs a DFT across its subcarriers before the OFDM modulation, effectively spreading the signal across a single-carrier waveform.

VoLTE Call Flow

Voice over LTE (VoLTE) is the standard mechanism for voice calls over LTE networks, using IMS (IP Multimedia Subsystem) to provide SIP-based signalling and RTP for voice media. This is fundamentally different from CSFB (Circuit-Switched Fallback), where the UE drops to 3G/2G for voice.

IMS Node Functions

  • P-CSCF Proxy-CSCF: First IMS contact point for the UE. Handles SIP signalling, interacts with PCRF for media authorization (Rx interface), provides SIP compression (SigComp).
  • I-CSCF Interrogating-CSCF: Acts as entry point for incoming calls to the operator's IMS. Queries HSS to determine serving S-CSCF.
  • S-CSCF Serving-CSCF: The central IMS signalling node for a registered UE. Performs registration, session control, service invocation (via ISC interface to application servers), and charging.
  • AS Application Server: Hosts IMS services (voicemail, call forwarding, conferencing, push-to-talk). Connected to S-CSCF via ISC (SIP) interface.
  • MGCF Media Gateway Control Function: Controls the Media Gateway (MGW) for interworking between IMS (VoLTE) and the PSTN/CS domain.

VoLTE Bearer Setup in Relation to IMS

The key interaction during VoLTE is between SIP signalling and the QoS bearer establishment. When the P-CSCF receives the SDP answer (with media parameters), it sends an Rx AA-Request to the PCRF with the media parameters. The PCRF correlates this with the Gx session and installs a PCC rule creating a dedicated QCI=1 bearer on the PGW. This dedicated bearer is established before the call connects (during the 183/180 exchange), ensuring media can flow immediately when the call is answered.

Roaming Architecture

LTE supports international roaming where a UE uses its home operator's subscription on a visited network. This involves specific interfaces and procedures to authorize the roamer and ensure correct billing.

Roaming Scenarios

LOCAL BREAKOUT

LBO Roaming

Data traffic exits to the internet through the Visited PLMN's PGW (vPGW). Lower latency for internet access. PCRF interaction via S9 interface between V-PCRF and H-PCRF. S8 interface (variant of S5) connects vSGW to vPGW.

HOME ROUTED

Home Routed Roaming

Data is tunneled all the way back to the Home PLMN's PGW (hPGW). Allows home operator full control and policy enforcement. S8 interface connects vSGW to hPGW across the PLMN boundary. Higher latency but simpler policy.

GRX/IPX — Roaming Interconnect

The GPRS Roaming Exchange (GRX), now evolved into IP eXchange (IPX), is the global interconnect network for mobile roaming signalling and data. Operators connect their S6a, S8, S9, S10, and Diameter Signalling Routers to the IPX network, enabling signalling to flow between any two connected operators' core networks worldwide.

Visited Network Authorization

The vMME verifies that the roaming UE's home PLMN is on the visited network's allowed roaming list (stored locally and synchronized with the home operator's roaming agreement). Subscription-level roaming restrictions (e.g., data roaming disabled by subscriber, or ODB — Operator Determined Barring) are enforced by the hHSS and communicated via the ULA response during attach.

Diameter Routing Agent (DRA): In large networks, Diameter Routing Agents sit between Diameter nodes to route messages efficiently, provide load balancing across multiple MMEs and HSSes, and handle inter-operator Diameter routing across the IPX/GRX. The DRA is defined at reference points Dh (to HSS) and Dm (between DRAs).

LTE System in Perspective

LTE represents one of the most sophisticated mobile communications systems ever deployed. Its all-IP design, flat architecture, advanced radio techniques (OFDMA, MIMO), and rich QoS framework enable a diverse range of services — from simple SMS-over-SGs to multi-gigabit carrier-aggregated downloads and HD VoLTE calls.

Every interaction — from the millisecond scheduling decisions in the eNB's MAC scheduler to the Diameter exchanges between MME and HSS, from the X2 handover executing in under 50 ms to the PCRF dynamically adjusting bearers for a new video call — represents an elegant interplay of dozens of protocols, nodes, and interfaces working in concert.

LTE continues to evolve: its physical layer techniques now form the foundation of 5G NR, its EPC is evolving toward the 5G Core (5GC) with even greater separation of control and user planes (via the CUPS architecture, and ultimately the 5G SBA — Service-Based Architecture), and its QoS model has been generalized and extended to support network slicing in 5G.

Understanding LTE thoroughly — at the level of every interface, every protocol, every call flow step — provides the foundation for understanding not just current networks, but the entire trajectory of mobile telecommunications into the 5G era and beyond.
