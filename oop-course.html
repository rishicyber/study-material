Opening · 20 min The Problem with "Open" Data
▼
📖 Analogy — Lead with this
The ATM Machine: When you use an ATM, you don't reach into its vault and grab money yourself. You interact through a controlled interface — PIN pad, screen, card slot. The machine's internal logic (balance checking, dispensing mechanism, transaction logging) is hidden from you. You get exactly what you need and nothing more. This is encapsulation.

Now ask the class: "What would happen if anyone could walk up and directly change the balance variable inside the bank's database?"
💥 Counter-example — Show the problem first
PYTHON — Bad Code (No Encapsulation)

        self.balance = balance   # PUBLIC — anyone can write to this!

account = BankAccount("Alice", 1000)

# Anyone can do this — no validation, no audit trail

account.balance = "hacked"  # 💀 wrong type, crashes later
account.owner   = None      # 💀 who owns this now?

Core Theory · 30 min Access Modifiers, Getters & Setters, Data Hiding
▼
🧠 Core Concepts
private
Accessible only within the class itself. The strongest protection. Use for internal state.
protected
Accessible within class and subclasses. Convention: _variable in Python.
public
Accessible anywhere. Use sparingly — only for things that truly need external access.
getter
A method that reads a private attribute. Can add validation, formatting, or computation.
setter
A method that writes a private attribute. Enforces rules before accepting a value.
@property
Python's Pythonic way — makes getters/setters look like attribute access.
✅ The Fix — Live Code This
PYTHON — Proper Encapsulation 
class BankAccount:
    def __init__(self, owner: str, balance: float):
        self.__owner   = owner        # private: __ prefix
        self.__balance = 0.0         # always start via setter for validation
        self.balance   = balance      # uses property setter below

    # ── GETTER ──────────────────────────────
    @property
    def balance(self) -> float:
        return self.__balance

    # ── SETTER (with validation) ──────────────
    @balance.setter
    def balance(self, amount: float):
        if not isinstance(amount, (int, float)):
            raise TypeError("Balance must be numeric")
        if amount < 0:
            raise ValueError("Balance cannot be negative")
        self.__balance = amount

    # ── BEHAVIOUR ────────────────────────────
    def deposit(self, amount: float) -> None:
        self.balance += amount  # goes through setter
        print(f"Deposited {amount}. New balance: {self.__balance}")

    def withdraw(self, amount: float) -> bool:
        if amount > self.__balance:
            print("Insufficient funds")
            return False
        self.__balance -= amount
        return True

# Usage
acc = BankAccount("Alice", 1000)
acc.deposit(500)          # ✅ Deposited 500. New balance: 1500
acc.balance = -999       # ✅ Raises ValueError — caught!
print(acc.__balance)      # ✅ AttributeError — truly private

Deep Dive · 30 min Real-World Design: Information Hiding & Tell, Don't Ask
▼
🌍 Real-World Application
"Tell, Don't Ask" Principle: Instead of asking an object for its data and making decisions outside, tell the object to do something and let it decide how. This is the soul of encapsulation in practice.

Bad: "Give me your age. If age > 18, you're eligible." — logic lives outside.
Good: "Are you eligible?" — object applies its own rules internally.
🔍 Java Example — with access modifiers
JAVA — Encapsulation with Tell-Don't-Ask 
public class Employee {
    private String  name;
    private double  baseSalary;
    private int     yearsOfService;

    public Employee(String name, double salary, int years) {
        this.name           = name;
        this.baseSalary     = validateSalary(salary);
        this.yearsOfService = years;
    }

    // Tell, don't ask — logic stays inside the class
    public double calculateAnnualBonus() {
        double rate = yearsOfService > 5 ? 0.15 : 0.05;
        return baseSalary * rate;
    }

    public boolean isEligibleForPromotion() {
        return yearsOfService >= 3 && baseSalary < 100000;
    }

    public void giveRaise(double percent) {
        if (percent < 0 || percent > 50)
            throw new IllegalArgumentException("Invalid raise %");
        baseSalary *= (1 + percent / 100);
    }

    private double validateSalary(double s) {
        if (s < 0) throw new IllegalArgumentException("Negative salary");
        return s;
    }

    // Only expose what callers truly need
    public String getName()        { return name; }
    public double getBaseSalary()  { return baseSalary; }
}

Core Theory · 30 min Abstract Classes, ABCs, and Contracts
▼
💻 Python Abstract Base Class
PYTHON — Abstract Class with abc module
from abc import ABC, abstractmethod

# ── The CONTRACT (abstract) ──────────────────────
class Shape(ABC):
    """Abstract shape. Defines WHAT every shape must do,
    not HOW it does it."""

    @abstractmethod
    def area(self) -> float:
        """Return the area of this shape."""

    @abstractmethod
    def perimeter(self) -> float:
        """Return the perimeter of this shape."""

    # Concrete method — shared behaviour
    def describe(self) -> str:
        return f"{type(self).__name__}: area={self.area():.2f}, perimeter={self.perimeter():.2f}"

# ── CONCRETE IMPLEMENTATIONS ─────────────────────
class Circle(Shape):
    def __init__(self, radius: float):
        self.__radius = radius

    def area(self) -> float:
        return 3.14159 * self.__radius ** 2

    def perimeter(self) -> float:
        return 2 * 3.14159 * self.__radius

class Rectangle(Shape):
    def __init__(self, w: float, h: float):
        self.__w, self.__h = w, h

    def area(self) -> float:
        return self.__w * self.__h

    def perimeter(self) -> float:
        return 2 * (self.__w + self.__h)

# ── CALLER doesn't care HOW — just calls the contract ─
shapes = [Circle(5), Rectangle(4, 6), Circle(3)]
for shape in shapes:
    print(shape.describe())   # Works for ALL shapes!

# Cannot instantiate abstract class:
s = Shape()   # ✅ TypeError — cannot instantiate abstract class

Deep Dive · 30 min Abstract Class vs Interface · Real Payment System
▼
⚖ Abstract Class vs Interface
Abstract Class
Can have concrete methods + state. Partial implementation. "Is-a" relationship. Single inheritance in Java.
Interface (Java)
Pure contract — only method signatures (before Java 8). Multiple interfaces allowed. "Can-do" relationship.
Python Protocol
Duck typing + structural subtyping. If it has the right methods, it works — no explicit inheritance needed.
🏦 Real System: Payment Gateway Abstraction
PYTHON — Payment System (Real-world abstraction)
from abc import ABC, abstractmethod
from dataclasses import dataclass

@dataclass
class PaymentResult:
    success: bool
    transaction_id: str
    message: str

class PaymentGateway(ABC):
    """Abstract gateway — any payment provider must implement this."""

    @abstractmethod
    def charge(self, amount: float, currency: str) -> PaymentResult:
        pass

    @abstractmethod
    def refund(self, transaction_id: str) -> PaymentResult:
        pass

# Each provider implements the same interface differently
class StripeGateway(PaymentGateway):
    def charge(self, amount, currency):
        # Stripe-specific API calls hidden here
        return PaymentResult(success=True, transaction_id="ch_stripe_123", message="OK")

    def refund(self, transaction_id):
        return PaymentResult(success=True, transaction_id=transaction_id, message="Refunded")

class PayPalGateway(PaymentGateway):
    def charge(self, amount, currency):
        return PaymentResult(success=True, transaction_id="pp_txn_456", message="OK")

    def refund(self, transaction_id):
        return PaymentResult(success=True, transaction_id=transaction_id, message="Refunded")

# ── OrderProcessor doesn't know which gateway it uses ─
class OrderProcessor:
    def __init__(self, gateway: PaymentGateway):  # depends on abstraction!
        self.__gateway = gateway

    def process_order(self, amount: float) -> str:
        result = self.__gateway.charge(amount, "INR")
        return result.transaction_id if result.success else "FAILED"

# Swap payment providers with zero changes to OrderProcessor!
processor = OrderProcessor(StripeGateway())
processor = OrderProcessor(PayPalGateway())  # same interface ✅

🔑 The Critical Distinction (write on board)
Encapsulation asks: "What should be hidden from outside users of this class?"

Abstraction asks: "What is the essential interface users need, regardless of how I implement it?"

They work together: Abstraction defines the what. Encapsulation protects the how.

Hook · 20 min Why Re-invent the Wheel? The Biology of Code
▼
📖 Opening Analogy
Biological Taxonomy: All mammals breathe air, nurse young, and are warm-blooded. A Dog doesn't re-implement "breathe" — it inherits it from Mammal. But Dog also overrides with specific behavior: it barks (Cat meows). The shared traits live in one place; specialists override only what differs.

In code: "Don't repeat yourself (DRY). Write common behavior once in a parent; let children specialize."
💻 The Foundation — Parent Class
PYTHON — Inheritance Fundamentals
class Vehicle:        # Parent (Base) class
    def __init__(self, make: str, model: str, year: int):
        self.make  = make
        self.model = model
        self.year  = year
        self._speed = 0          # protected — subclasses can access

    def accelerate(self, amount: int) -> None:
        self._speed += amount
        print(f"{self.model} speed: {self._speed} km/h")

    def fuel_type(self) -> str:     # to be overridden
        return "unknown"

    def __str__(self) -> str:
        return f"{self.year} {self.make} {self.model} ({self.fuel_type()})"


class Car(Vehicle):    # Inherits everything from Vehicle
    def __init__(self, make, model, year, num_doors: int):
        super().__init__(make, model, year)   # call parent __init__
        self.__num_doors = num_doors

    def fuel_type(self) -> str:     # OVERRIDE
        return "petrol"


class ElectricCar(Car):  # Multi-level: ElectricCar → Car → Vehicle
    def __init__(self, make, model, year, num_doors, battery_kwh: float):
        super().__init__(make, model, year, num_doors)
        self.__battery_kwh = battery_kwh

    def fuel_type(self) -> str:     # OVERRIDE again
        return f"electric ({self.__battery_kwh} kWh battery)"

    def charge(self) -> None:        # NEW method unique to ElectricCar
        print(f"Charging {self.model}...")

# All vehicles respond to the same interface:
vehicles = [
    Car("Toyota", "Camry", 2022, 4),
    ElectricCar("Tesla", "Model 3", 2023, 4, 82),
]
for v in vehicles:
    print(v)         # __str__ inherited by all
    v.accelerate(30) # method also inherited

Advanced · 15 min MRO, Liskov Substitution Principle & The Fragile Base Problem
▼
🔬 Python MRO — Method Resolution Order
PYTHON — MRO and Diamond Problem
class A:
    def hello(self): print("Hello from A")

class B(A):
    def hello(self): print("Hello from B")

class C(A):
    def hello(self): print("Hello from C")

class D(B, C):   # Multiple inheritance — diamond!
    pass

d = D()
d.hello()    # Which hello() wins?

print(D.__mro__)
# → (D, B, C, A, object)  ← Python uses C3 linearisation
# Left-to-right depth-first, but A appears only once
⚖ Liskov Substitution Principle (LSP)
LSP: If S is a subtype of T, then objects of type T may be replaced by objects of type S without breaking the program.

Simple test: Can you swap a Car for any place expecting a Vehicle and have everything still work? If yes — good inheritance. If no — you're breaking LSP.

Classic violation: Making Square inherit from Rectangle. A Square that overrides set_width() to also change height breaks the contract that Rectangle's sides are independent.

Critical Thinking · 25 min Composition vs Inheritance — When Inheritance Goes Wrong
▼
🏗 Composition Alternative
PYTHON — Composition (often better than deep inheritance)
# Instead of: FlyingSwimmingDuck → Duck → Animal (messy!)
# Compose behaviors as objects:

class FlyBehavior:
    def fly(self): print("Flying with wings")

class NoFlyBehavior:
    def fly(self): print("Cannot fly")

class Duck:
    def __init__(self, name: str, fly_behavior):
        self.name = name
        self.__fly = fly_behavior  # behavior COMPOSED in, not inherited

    def perform_fly(self):
        self.__fly.fly()

    def set_fly_behavior(self, behavior):  # can swap at RUNTIME!
        self.__fly = behavior

mallard = Duck("Mallard", FlyBehavior())
rubber  = Duck("Rubber",  NoFlyBehavior())

mallard.perform_fly()   # Flying with wings
rubber.perform_fly()    # Cannot fly
🎯 Key insight: Inheritance is "is-a". Composition is "has-a". Both are valid. The question is: does the subtype truly pass the Liskov test?
📏 Rule: Prefer composition when behavior varies independently, or when inheritance would require inheriting unwanted methods.

📋 The 3 Rules of Good Inheritance
✅ Rule 1 — Is-A test: SubClass IS-A ParentClass must be true in plain English. "A Dog is an Animal" ✓. "A Stack is a Vector" ✗ (Java's mistake).
✅ Rule 2 — LSP: Every subclass instance must be usable wherever a parent is expected, without breaking behavior.

Hook · 20 min The '+' Operator: Polymorphism You Already Know
▼
📖 Start with what they already know
The + Operator: Write this on the board:

2 + 3 = 5   (integer addition)
"hello" + " world" = "hello world"   (string concatenation)
[1,2] + [3,4] = [1,2,3,4]   (list joining)

Same operator +. Three different types. Three different behaviors. Python decides at runtime which behavior to invoke based on the object's type. This is polymorphism.
💻 Runtime Polymorphism — Core Example
PYTHON — Runtime Polymorphism via Method Overriding
class Animal:
    def __init__(self, name: str):
        self.name = name

    def speak(self) -> str:
        raise NotImplementedError("Subclass must implement speak()")


class Dog(Animal):
    def speak(self) -> str:
        return f"{self.name} says: Woof!"

class Cat(Animal):
    def speak(self) -> str:
        return f"{self.name} says: Meow!"

class Duck(Animal):
    def speak(self) -> str:
        return f"{self.name} says: Quack!"

# ── THE MAGIC ─────────────────────────────────────
# One loop. One method call. Different behavior each time.
animals = [Dog("Rex"), Cat("Whiskers"), Duck("Donald"), Dog("Buddy")]

for animal in animals:
    print(animal.speak())   # Python dispatches to right speak() at runtime

# Rex says: Woof!
# Whiskers says: Meow!
# Donald says: Quack!
# Buddy says: Woof!

# Adding a new animal requires ZERO changes to the loop — Open/Closed!
🎯 Key moment: Ask students: "How many lines change if we add a Parrot?" (Answer: just the Parrot class.) This is the power of polymorphism.

Python Special · 15 min Duck Typing & Operator Overloading
▼
🦆 Duck Typing
"If it walks like a duck and quacks like a duck, it's a duck."

In Python, you don't need a class hierarchy for polymorphism. If an object has the right method, Python will call it — regardless of its type or parent class. This is structural typing.
PYTHON — Duck Typing + Operator Overloading
# Duck typing — no inheritance needed!
class Robot:
    def speak(self) -> str:
        return "BEEP BOOP"

# Robot is not an Animal — but the loop doesn't care!
things = [Dog("Rex"), Robot()]
for t in things:
    print(t.speak())   # works for both — duck typing!

# ── OPERATOR OVERLOADING ──────────────────────────
class Vector:
    def __init__(self, x: float, y: float):
        self.x = x
        self.y = y

    def __add__(self, other: 'Vector') -> 'Vector':   # v1 + v2
        return Vector(self.x + other.x, self.y + other.y)

    def __mul__(self, scalar: float) -> 'Vector':    # v * 3
        return Vector(self.x * scalar, self.y * scalar)

    def __eq__(self, other) -> bool:                  # v1 == v2
        return self.x == other.x and self.y == other.y

    def __repr__(self) -> str:
        return f"Vector({self.x}, {self.y})"

v1 = Vector(1, 2)
v2 = Vector(3, 4)
print(v1 + v2)     # Vector(4, 6) — feels natural!
print(v1 * 3)     # Vector(3, 6)

🎓 Grand Lab Exercise — All 4 Pillars Together
Exercise: University Course Registration System
Build this step by step — each step requires a different pillar.
  1. [Encapsulation] Create a Course class: private _students list, private _capacity. Add enroll(student) with validation, get_enrollment_count().
  2. [Abstraction] Create an abstract Person base with abstract get_role() and concrete get_id_card() method.
  3. [Inheritance] Create Student(Person) with student_id, courses list, enroll_in(course). Create Professor(Person) with courses_teaching, assign_to(course).
  4. [Polymorphism] Create a Registrar class that holds a list of Person objects and can call get_role() on each — without knowing if they're a Student or Professor.
  5. Operator overloading: Override __str__ on all classes for clean printing. Override __len__ on Course to return enrollment count.
  6. Final test: Create 2 professors, 5 students, 3 courses. Enroll students. Print the registrar's full roster using a single polymorphic loop.
📐 Reference — All 4 Pillars in One View
Encapsulation
Course._students is private. Only enroll() can add, with validation. You can't corrupt the list directly.
Abstraction
Person defines get_role() as a contract. Registrar doesn't know the concrete type — it only knows the interface.
Inheritance
Student and Professor inherit from Person — they get get_id_card() for free and override get_role().
Polymorphism
The registrar loop calls person.get_role() — for each person, the right implementation runs automatically.

🏛 The Four Pillars — Final Summary
I Encapsulation — Protect. Bundle data with its behavior. Control access. Validate at the boundary.
II Abstraction — Simplify. Define contracts. Hide complexity. Program to interfaces.
III Inheritance — Reuse. Build on existing classes. Override what differs. Respect LSP.
IV Polymorphism — Extend. One interface, many behaviors. Add without modifying. Open/Closed.
